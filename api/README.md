# API Definitions

This directory contains the source OpenAPI specifications for GizClaw HTTP APIs.
These JSON files are the contract used by generated Go clients, server
interfaces, and shared API types under `pkgs/gizclaw/api/`.

## Layout

- `http/admin.json`, `http/peer.json`, and `http/desktop.json` define GizClaw
  HTTP API surfaces.
- `http/openai-compat/v1/service.json` defines the OpenAI-compatible HTTP
  surface.
- `http/shared.json` collects shared HTTP schemas and exposes them through
  `#/components/schemas`.
- `http/shared/*.json` contains reusable shared schema definitions.
- `http/shared/server.json` contains peer-owned DTO schemas still referenced by
  Admin HTTP generation.
- `http/resources/*.json` contains declarative admin resource schemas used by
  `admin apply`, `admin show`, and related resource APIs.
- `proto/rpc/rpc.proto` defines Peer RPC envelopes, errors, stream frames, and
  method registry.
- `proto/rpc/payload/*.proto` defines method-specific Peer RPC payload messages
  by domain.
- `proto/telemetry/peer_telemetry.proto` defines telemetry protobuf messages.

## Generated And Committed Code

Generated Go code lives outside this directory:

- `pkgs/gizclaw/api/adminhttp/generated.go`
- `pkgs/gizclaw/api/apitypes/generated.go`
- `pkgs/gizclaw/api/openaihttp/generated.go`
- `pkgs/gizclaw/api/rpcproto/rpc.pb.go`
- `pkgs/gizclaw/api/rpcproto/{ai,edge,enums,firmware,gameplay,social,system,workspace}.pb.go`
- `pkgs/gizclaw/api/peerhttp/generated.go`

`rpcproto` is generated directly by `protoc-gen-go` from
`api/proto/rpc/rpc.proto` and the split `api/proto/rpc/payload/*.proto` files.
`adminhttp`, `apitypes`, `openaihttp`, and `peerhttp` are generated by
`oapi-codegen`; the repository-owned `apitypesgen` command resolves the schema
input but does not own the generated Go templates.

`rpcapi/generated.go` and `rpcapi/payload_codec.go` are legacy committed wrapper
code on top of the protobuf descriptors. There is currently no command that
regenerates them: `go generate ./pkgs/gizclaw/api/rpcapi` only validates the
package with tests. When changing Peer RPC methods or payload messages, update
the wrapper and codec deliberately and verify them with
`go test ./pkgs/gizclaw/api/rpcapi`. Handwritten APIs use the defining
`rpcpb` message types directly rather than adding compatibility aliases.

Current generated TypeScript SDK code lives under `sdk/js/gizclaw/`:

- `sdk/js/gizclaw/generated/adminhttp`
- `sdk/js/gizclaw/generated/rpc`
- `sdk/js/gizclaw/generated/peerhttp`

The old browser `client_service` API and CLI-served UI TypeScript clients were
removed as part of the desktop clean break. Desktop UI code should consume the
generated clients through `@gizclaw/gizclaw` WebRTC transports instead of
reintroducing old CLI-served package boundaries.

Do not edit third-party generated files by hand. Change the source schema in
`api/`, then regenerate the corresponding Go and/or TypeScript package. The
legacy `rpcapi` wrapper is manually maintained and must stay consistent with the
source proto, generated `rpcproto` messages, and codec tests.

Common commands:

```sh
go generate ./pkgs/gizclaw/api/adminhttp
go generate ./pkgs/gizclaw/api/apitypes
go generate ./pkgs/gizclaw/api/openaihttp
go generate ./pkgs/gizclaw/api/rpcproto
go generate ./pkgs/gizclaw/api/peerhttp
go test ./pkgs/gizclaw/api/rpcapi
```

Regenerate TypeScript SDK packages with:

```sh
npm --prefix sdk/js run gen:sdk
```

When in doubt, regenerate all API packages:

```sh
go generate ./pkgs/gizclaw/api/...
```

## Maintenance Guidelines

- Treat files in `api/` as public contracts. Keep changes small, explicit, and
  covered by tests at the service or CLI boundary.
- Prefer adding genuinely shared schemas under `http/shared/` and resource-only
  schemas under `http/resources/`. If a type is used by only one resource,
  embed it in that resource schema instead of promoting it to the shared layer.
- Keep schema names, discriminator values, and path operation IDs stable unless
  the caller-facing contract is intentionally changing.
- When adding or changing an endpoint, update the OpenAPI document, regenerate
  Go code, implement the strict server interface, and add tests for both success
  and user-visible error paths.
- When changing declarative admin resources, verify `resourcemanager` behavior
  and CLI stories under `tests/gizclaw-e2e/cmd/` when applicable.
- Run focused tests for the touched API surface and coverage-sensitive packages.
  For broader API changes, prefer:

```sh
go test ./pkgs/gizclaw/... ./cmd/... -count=1
```
